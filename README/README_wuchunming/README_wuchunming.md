于是，我们就开始写控制算法。我们决定将横向速度与纵向速度分开控制。横向速度使用纯跟踪算法，纵向速度使用PID算法控制。
对于横向速度来说，我们先选择一个前瞻距离，找到路径中最接近这个前瞻距离的那个前瞻点，然后计算车的当前位置到前瞻点的横向距离，最后通过公式来计算出所需要的转向角。
我们找这个前瞻点的方法，就是在路径上寻找距离最接近前瞻距离且在车辆前方的点。同时，我们应该在上一瞬间的最近点局部做搜索，防止找到之前路径上的点，让小车绕回去。
我们还有一个绝妙的想法，对前瞻距离的选择做了优化。我们使用公式`ld = max(min_lookahead, lookahead_distance + current_speed * lookahead_ratio)`实现了基于当前速度的自适应前瞻距离。小车速度越快，前瞻距离就越大，防止小车一下就冲出了轨道。

对于纵向速度的PID算法，我们的想法是，根据当前规划路径的曲率来决定当前的参考速度，路径越弯的地方，我们就降低参考速度，防止车出去。曲率使用三点法计算，然后PID去跟踪这个参考速度。为了防止参考速度随路径曲率突变，进而导致微分项暴涨，我们使用了微分先行的思想，只对我们得到的当前速度求导。由于这是个不会突变的量，可以保证其微分有限。同时，我们还加上了积分上限，防止加速度达到最大时积分继续增长，难以下降。

最后，我们再写一个主要的控制节点，用来读取车的位姿信息和规划的路径、计算曲率、检测停车以及把横向控制得到的转向角与线速度合成为twist信息并发布。检测停车的办法，是计算得出路径的切线，然后看终点到车坐标的向量在切线上的投影，如果超过了终点，就把速度设为0。

后来，我们去问了AI，AI给了我们一些建议。它认为我们偏离轨道那么多，一定是因为修正角还不够大所导致的。于是，它让我们根据当前的横向偏差CTE来修正转向角，让他变得更大。显然，这肯定是错的，现在的偏差是由于修正角已经太大了才导致的。但我们没有意识到这一点，被蒙在鼓里好久。
同时，它还觉得我们虽然有了微分优先，但突然变化的参考速度还是会产生影响。它让我们给计算出来的路径曲率进行滤波

这期间，我看旧的路行不通，就想去试一试MPC的方法，正好在上个作业里面有讲到这个东西。MPC的算法是把车的各种状态表达成向量，建立向量空间转移方程，预测接下来n步。将每一次转移之后的向量与目标位置构建误差函数，最后用求解这个误差函数值最小处的，用来空间转移的向量。这一部分具体的代码是AI写的，因为我们也没有预期把它作为最后的控制方法。在实现上，我们只是设立了几种未来的转向角与线速度的组合，用来预测并求误差函数，最后找到误差函数最小时的转向角与角速度。虽然只是MPC青春版，但运行效果还不错。

言归正传。后来，组内的其他同学也陆陆续续地优化了感知部分和规划部分的代码。现在锥桶的位置不会因为噪声再时时改变了，所以规划出来的路径也稳定了。我们也能更方便地进行下一步优化。
我们发现，只使用现在的停车逻辑还是不能停好车，于是我们又写了更多的逻辑（尸米山）用来停车。当车没有收到新的路径超过两秒就停车，当车离终点只剩一定距离时就减速。然后，我们终于发现了这个byd的横向误差修正。由于改完之后的公式实在跟原来的公式差得有点多，我们就寻思把它扔了吧，这样还能减少转向角。没想到一下就成了。
